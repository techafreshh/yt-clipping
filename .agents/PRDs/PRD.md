# Product Requirements Document: YouTube Shorts Repurposing Workflow

## 1. Executive Summary

The YouTube Shorts Repurposing Workflow is a CLI tool that transforms long-form YouTube content into vertical shorts (9:16, 1080×1920). It takes two synced video perspectives of the same recording — a camera view and a screen view — identifies clip-worthy moments, cuts both tracks, and composites them into a stacked vertical format ready for upload to YouTube Shorts, TikTok, and Reels.

The tool supports two modes of highlight identification: manual timestamps provided by the creator, and AI-suggested highlights generated by an LLM (via OpenRouter) analyzing the video's transcript (fetched via Supadata API or provided locally). This dual-mode approach gives creators full control when they know exactly what they want, while offering AI assistance for discovering moments they might miss.

**MVP Goal:** Deliver a working CLI tool that accepts two synced video files, identifies clips via manual timestamps or AI suggestions, and outputs production-ready vertical shorts with configurable layout and optional TikTok-style captions — all powered by Python and ffmpeg with no heavy dependencies.

---

## 2. Mission

**Mission Statement:** Enable YouTube creators to efficiently repurpose their long-form content into high-quality vertical shorts, maximizing content reach across platforms with minimal manual effort.

**Core Principles:**

1. **Speed over perfection** — Batch-process clips quickly using ffmpeg stream-copy where possible; re-encode only when necessary.
2. **Creator control** — Every automated decision (clip selection, layout, captions) can be overridden or configured.
3. **Minimal dependencies** — Python + ffmpeg + HTTP APIs. No heavy ML models running locally, no moviepy.
4. **Convention over configuration** — Sensible defaults (50/50 split, 30–60s clips, no captions) that work out of the box with a simple folder structure.
5. **Composable pipeline** — Each step (transcript → highlights → cut → composite) is independently runnable and debuggable.

---

## 3. Target Users

### Primary Persona: Solo YouTube Creator

- **Profile:** Runs a YouTube channel with long-form content (tutorials, coding sessions, commentary) recorded with dual perspectives (camera + screen capture).
- **Technical comfort:** Comfortable with CLI tools, Python, and basic terminal usage. Not a professional video editor.
- **Pain points:**
  - Manually editing shorts in video editors is time-consuming and repetitive.
  - Identifying the best moments across a 1–2 hour recording is tedious.
  - Maintaining consistent output format (aspect ratio, layout, captions) across many shorts requires discipline.
- **Needs:**
  - Fast batch production of shorts from existing recordings.
  - AI assistance in finding clip-worthy moments.
  - Consistent, platform-ready output without manual video editing.

## 5. User Stories

### Primary User Stories

1. **As a creator, I want to provide manual timestamps for clips**, so that I can quickly produce shorts from moments I've already identified while watching my content.
   - *Example:* Drop `clips/episode12.json` with `[{start: "00:12:34", end: "00:13:25", slug: "best-tip"}]`, run `shorts cut episode12`, get a finished short.

2. **As a creator, I want AI to suggest clip-worthy moments from my transcript**, so that I can discover engaging segments I might have missed.
   - *Example:* Run `shorts suggest episode12 --model anthropic/claude-sonnet-4 --count 5` and get 5 proposed clips with timestamps, slugs, and hooks.

3. **As a creator, I want to fetch my video's transcript automatically via its YouTube URL**, so that I don't need to manually transcribe or export captions.
   - *Example:* Run `shorts transcript episode12 --youtube-url https://youtube.com/watch?v=...` and get `transcripts/episode12.json`.

4. **As a creator, I want my shorts to be vertical 9:16 with screen on top and camera on bottom**, so that they're optimized for YouTube Shorts and TikTok without manual editing.
   - *Example:* Output is always 1080×1920 with the two views stacked.

5. **As a creator, I want to configure the layout split between screen and camera**, so that I can emphasize whichever view is more relevant for a given video.
   - *Example:* `shorts cut episode12 --split 60` gives 60% screen, 40% camera.

6. **As a creator, I want optional TikTok-style captions burned into my shorts**, so that I can improve retention without using a separate captioning tool.
   - *Example:* `shorts cut episode12 --captions` produces shorts with synced word-by-word highlighted text.

7. **As a creator, I want to run the entire pipeline end-to-end with a single command**, so that I can go from raw recordings to finished shorts with minimal steps.
   - *Example:* `shorts run episode12 --youtube-url <url> --captions` does transcript → suggest → cut → output.

### Technical User Stories

8. **As a developer, I want each pipeline step to be independently runnable**, so that I can debug issues at any stage without re-running the entire workflow.
   - *Example:* Fix a bad clip timestamp in `clips/episode12.json` and re-run only `shorts cut episode12` without re-fetching the transcript.

## 6. Core Architecture & Patterns

### High-Level Architecture

The tool follows a **pipeline architecture** with discrete, composable stages:

```
Input (raw/) → Transcript → Highlights → Cut → Composite → Output
```

Each stage reads from and writes to the filesystem, making the pipeline resumable and debuggable at any point.

### Directory Structure

```
yt shorts/
├── raw/                          # Input videos
│   ├── {source_name}_camera.mp4
│   └── {source_name}_screen.mp4
├── transcripts/                  # Cached transcripts
│   └── {source_name}.json
├── clips/                        # Highlight specs (manual or AI-generated)
│   └── {source_name}.json
├── output/                       # Final shorts
│   └── {source_name}/
│       └── {source_name}_short_01_{slug}.mp4
├── src/shorts/
│   ├── __init__.py
│   ├── config.py                 # Env + CLI config
│   ├── transcript.py             # Supadata fetch + file loader
│   ├── highlights.py             # OpenRouter suggestions + manual parser
│   ├── cutter.py                 # ffmpeg cut both tracks
│   ├── compositor.py             # 9:16 stack with configurable split
│   ├── captions.py               # Optional TikTok-style caption burn
│   ├── pipeline.py               # Orchestration
│   └── cli.py                    # Typer CLI entry point
├── tests/
├── .env.example
├── pyproject.toml
└── README.md
```

### Key Design Patterns

- **File-based intermediates:** Each stage persists its output to disk (JSON for transcripts/clips, mp4 for video). This enables resumability and manual inspection/editing between stages.
- **Pydantic models for data contracts:** `Transcript`, `Clip`, and config models enforce schema validation at boundaries.
- **Subprocess for ffmpeg:** Direct `subprocess.run()` calls with constructed argument lists — no wrapper libraries. Enables full control over ffmpeg flags.
- **Fail-forward batch processing:** Per-clip errors are logged and skipped; the batch continues and reports a summary at the end.

## 7. Tools/Features

### Feature 1: Transcript Fetching

| Aspect | Detail |
|--------|--------|
| **Purpose** | Obtain timestamped transcript for highlight analysis |
| **Operations** | Fetch from Supadata API via YouTube URL; Load from local file |
| **Output** | `transcripts/{source_name}.json` with timestamped segments |
| **Key features** | Caches to disk; supports word-level timestamps when available; handles both JSON and plain text input |

### Feature 2: Highlight Selection

| Aspect | Detail |
|--------|--------|
| **Purpose** | Identify 30–60s clip-worthy moments |
| **Operations** | Parse manual `clips.json`; Generate AI suggestions via OpenRouter |
| **Output** | `clips/{source_name}.json` — array of `{start, end, slug, hook}` |
| **Key features** | Configurable model; configurable clip count; validates timestamps against source duration; slug auto-generation |

### Feature 3: Video Cutting

| Aspect | Detail |
|--------|--------|
| **Purpose** | Extract matching segments from both camera and screen tracks |
| **Operations** | ffmpeg `-ss`/`-to` on both files; stream-copy when possible |
| **Key features** | Audio retained on camera only; screen track stripped of audio; duration verification between both cuts |

### Feature 4: 9:16 Compositing

| Aspect | Detail |
|--------|--------|
| **Purpose** | Stack screen (top) and camera (bottom) into vertical output |
| **Operations** | ffmpeg scale + pad + vstack filter chain |
| **Output** | `output/{source_name}/{source_name}_short_{NN}_{slug}.mp4` |
| **Key features** | Configurable split ratio; H.264/AAC output; 1080×1920 guaranteed; black bars for aspect ratio mismatch |

### Feature 5: TikTok-Style Captions

| Aspect | Detail |
|--------|--------|
| **Purpose** | Burn synced captions into shorts for retention |
| **Operations** | Generate ASS subtitle file from transcript timestamps; apply via ffmpeg subtitles filter |
| **Key features** | Word-by-word highlight; bold white + black stroke; centered; falls back to segment-level when word timestamps unavailable |

### Feature 6: End-to-End Pipeline

| Aspect | Detail |
|--------|--------|
| **Purpose** | Single-command workflow from raw files to finished shorts |
| **Operations** | Orchestrates: transcript → suggest → cut → composite (→ captions) |
| **Key features** | Progress logging; partial failure handling; configurable via CLI flags |

## 9. Security & Configuration

### Configuration Management

All sensitive configuration via `.env` file:

```env
SUPADATA_API_KEY=your_supadata_key
OPENROUTER_API_KEY=your_openrouter_key
DEFAULT_MODEL=anthropic/claude-sonnet-4
DEFAULT_SPLIT=50
DEFAULT_CLIP_DURATION_MIN=30
DEFAULT_CLIP_DURATION_MAX=60
```

### Security Scope

**In scope:**
- API keys stored in `.env` (gitignored)
- No secrets logged or printed (masked in `shorts config` output)
- Input validation on all user-provided timestamps and file paths

**Out of scope:**
- Multi-user authentication (single-user CLI tool)
- Network security (local tool, outbound HTTPS only)
- Video content DRM or access control

### Deployment

- Local installation via `pip install -e .` or `pip install .`
- Requires ffmpeg installed and available on PATH
- No containerization needed for MVP

## 10. API Specification

### External APIs Consumed

#### Supadata Transcript API

```
GET https://api.supadata.ai/v1/transcript?url={youtube_url}
Headers: x-api-key: {SUPADATA_API_KEY}

Response 200:
{
  "content": [
    { "start": 0.0, "end": 4.5, "text": "Hey everyone..." },
    ...
  ]
}
```

#### OpenRouter Chat Completions

```
POST https://openrouter.ai/api/v1/chat/completions
Headers: Authorization: Bearer {OPENROUTER_API_KEY}

Request:
{
  "model": "anthropic/claude-sonnet-4",
  "messages": [
    { "role": "system", "content": "..." },
    { "role": "user", "content": "{transcript_text}" }
  ],
  "response_format": { "type": "json_object" }
}

Response 200:
{
  "choices": [{
    "message": {
      "content": "{\"clips\": [{\"start\": \"00:12:34\", \"end\": \"00:13:25\", \"slug\": \"best-tip\", \"hook\": \"The one thing...\"}]}"
    }
  }]
}
```

### Internal Data Formats

#### Clip Spec (`clips/{source_name}.json`)

```json
[
  {
    "start": "00:12:34",
    "end": "00:13:25",
    "slug": "best-tip",
    "hook": "The one thing nobody tells you about..."
  }
]
```

#### Transcript (`transcripts/{source_name}.json`)

```json
{
  "segments": [
    { "start": 0.0, "end": 4.5, "text": "Hey everyone welcome back" },
    { "start": 4.5, "end": 8.2, "text": "today we're going to talk about..." }
  ],
  "words": [
    { "start": 0.0, "end": 0.3, "text": "Hey" },
    { "start": 0.3, "end": 0.8, "text": "everyone" }
  ]
}
```

## 11. Success Criteria

### MVP Success Definition

The tool is successful when a creator can go from two raw video files to a folder of upload-ready vertical shorts in under 5 minutes of hands-on time (excluding processing).

### Functional Requirements

- [x] Produces valid 1080×1920 H.264/AAC mp4 files playable on all platforms
- [x] Audio is present and synced in output (from camera track)
- [x] Manual timestamps mode produces clips at exact specified times
- [x] AI suggestion mode returns relevant, self-contained clip proposals
- [x] Configurable split ratio correctly adjusts layout proportions
- [x] Captions, when enabled, are synced to speech and styled correctly
- [x] Batch processing completes even if individual clips fail
- [x] CLI provides clear error messages for missing files, bad config, or API failures

### Quality Indicators

- Processing time per clip: < 30 seconds for a 60s clip on modern hardware
- No audio drift or desync in output
- Output file sizes reasonable (< 50MB per 60s clip at 1080p)
- Zero manual ffmpeg commands needed by the user

### User Experience Goals

- First-run setup in under 5 minutes (install deps, set `.env`, drop files)
- Single command for full pipeline (`shorts run`)
- Clear progress feedback during batch processing
- Helpful error messages that suggest fixes

## 14. Risks & Mitigations

| # | Risk | Impact | Mitigation |
|---|------|--------|------------|
| 1 | **ffmpeg keyframe misalignment** — Stream-copy cuts may start on wrong keyframe, causing brief black frames or glitches | Medium | Detect keyframe distance; fall back to re-encode (`-c:v libx264`) when cut point is far from nearest keyframe |
| 2 | **Supadata API unavailability or rate limits** | Medium | Support local transcript file as fallback; cache all fetched transcripts to disk; implement retry with backoff |
| 3 | **LLM returns malformed or out-of-range timestamps** | Medium | Validate all AI-suggested clips against source duration; retry once on parse failure; surface clear error with the raw response for debugging |
| 4 | **Source videos have mismatched durations despite same start** | Low | Warn if durations differ by >1s; use the shorter duration as the upper bound for clip end times |
| 5 | **Word-level timestamps unavailable from Supadata** | Low | Gracefully fall back to segment-level captions; warn user that word-highlight style requires word timestamps |
